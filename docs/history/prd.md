需求是这样，背景是我现在是在飞书上跟openclaw聊天吧。但是其实飞书的限制挺多的，比如说机器人与机器人之间不能聊天。再一个就是，其实我不太愿意绑定在某个聊天平台上，所以我想自己写一个。

所以首先说一下要实现的功能：
1. 网页端即时通信
    要在网页上能够实现我与OpenClaw机器人的即时通信聊天，能支持文本、图片显示。
2. 语音接收
    能支持我这边听语音（不需要支持我发语音给你）。
3. 群组支持
    支持群组，因为我需要把你们拉到一个群里面。
4. 通信模式
    模式跟飞书很像，OpenClaw和我的IM服务器之间走Socket建立长连接，实现实时沟通。
    要支持适配层，因为我还想接入nanobot以及其他agent。
5. 部署与认证
    我会部署在外网，有公网IP地址，OpenClaw可以直接连上去。连接建立需要认证，防止恶意链接。
6. 界面与多端适配
    界面稍微好看一点，同时支持在PC端和手机端能正常展示。

关于技术选型：
1. 后端
    后端语言我比较喜欢用Python，框架用Flask，存储用SQLite。尽量不要引入Redis，因为服务器内存不太够。
2. 前端
    前端用最普通的HTML+CSS+JS来写，因为我对React不是很熟。

后续流程：
1. 部署与安全
    我会把你放在nginx后面，不用担心HTTPS的问题，你正常写就行。
2. 自测与调试
    你写完之后要自己做一下库的review和功能验收。如果一遍写的有问题，就继续调试，直到不出问题为止。
3. 环境隔离
    调试时使用虚拟环境（Python -m venv），不要影响本机的包管理，避免把整个环境搞乱

OpenClaw 部署在本地或内网机器上，由这台机器主动连接到我们的公网 IM 服务器。

在消息能力上，希望支持常见的 IM 消息类型，包括：

- 文本消息
- 文件
- 视频
- 代码块
- Markdown 渲染

整体体验尽量对齐 Feishu 这类 IM 产品。

**关于语音消息，设想如下：**

1. Agent 直接生成 MP3 或 WAV 音频文件
2. 将音频上传到我们的 IM 服务器
3. 服务器侧存储并下发为语音消息

历史消息的默认保留时间，建议先定为 **30 天**。

---

还有一个需要确认的点是 Agent 与 IM 服务器之间的 WebSocket 协议。

我目前不确定 OpenClaw 是否已经有一套成熟的默认协议。例如它在接入 Feishu 等渠道时，是否已具备现成的消息收发和事件通信机制。

这里建议先查一下：

- OpenClaw 是否已经有默认的 WebSocket / Channel 接入协议？
- 这套协议是否已覆盖消息收发、文件消息、语音消息、事件回调等基础能力？

如果已经有成熟方案，应尽量直接复用其默认协议，而不是自己重新设计一套。
